### [1] 에셋 카탈로그 데이터 반영 지연 단축 옵션

**현재 동작 (v1.8.x 기준):**

| 데이터 | 반영 지연 (최악) |
|--------|-----------------|
| 신규 카테고리 / 에셋 (catHier / objects.json 추가) | ~2시간 (서버 1h 캐시 + 클라이언트 1h 폴링) |
| 기존 에셋 필드 수정 (설명·태그·가격) | ~2시간 (동일) |
| 기존 id 의 아이콘 PNG 교체 | 최대 24시간 (브라우저 이미지 캐시) |
| 카탈로그 iframe 상세 (`:8080/#item=…`) | 즉시 (매번 라이브 로드) |

**즉시 반영 수단:**
- `GET /api/object-meta?force=1` → 서버 캐시 무효화
- 브라우저 `Ctrl + Shift + R` → 이미지 캐시 포함 전체 리로드

**개선 옵션 (필요 시):**

1. **서버 캐시 주기 단축** — `server.js` 의 `META_CACHE_TTL` 을 1h → 5min.
   트레이드오프: upstream(`:8080`) fetch 빈도 12배 증가, 여전히 ms 단위라 부담 거의 없음.
2. **클라이언트 폴링 주기 단축** — `inzoi-concept-tool.jsx` 의 object-meta useEffect
   `setInterval(load, 60*60*1000)` 을 `5*60*1000` 으로.
3. **아이콘 캐시 완화** — `/api/object-icon/:id` 의 `cache-control: public, max-age=86400`
   을 `no-cache` 또는 `max-age=60` 으로. 트레이드오프: 동일 아이콘 반복 로드 대역폭↑.
4. **웹소켓 / SSE 알림** — inzoiObjectList 쪽에서 파일 변경 감지해 concept-studio 에
   즉시 갱신 신호. 양쪽 수정 필요, 과투자일 가능성.

**결정 필요:** 현재 주기로 현업 불편 보고가 나오면 옵션 1+2 부터 적용. 아이콘 교체
주기가 짧다면 옵션 3.

### [2] 기존 제작 에셋 시각 유사도 기반 Top-N 추천 (Option C)

**현재:** 기존 제작 에셋 썸네일 그리드는 카테고리 내 **customize/unlockable 아닌 순**
으로 상위 8개 고정 노출. 현재 카드 이미지와의 실제 유사도는 반영 안 됨.

**목표:** 사용자가 만든 컨셉 이미지와 가장 비슷한 기존 에셋 top-5~8 추천.

**구현 스케치:**
1. 서버 측 일회성 job: `inzoiObjectList/data/objects.json` 5,265개 에셋 아이콘을 Gemini
   Vision 임베딩 (또는 CLIP 등) 으로 인덱싱 → `data/catalog_embeddings.json` 에 저장.
2. 카드의 참조이미지 또는 선정 시안을 같은 방식으로 임베딩.
3. 코사인 유사도 top-N 계산해 `spec.sample_thumbs` 대신 유사도 순으로 반환.
4. UI 에 "🎯 유사도 NN%" 배지 추가.

**고려할 점:**
- Gemini 가 공식 임베딩 API 를 제공하는지 확인 필요 (Gemini 3 는 아직 이미지
  임베딩 API 가 제한적). 대안: Vertex AI Vision API, 또는 PIL/OpenCV + pHash 기반
  경량 유사도.
- 5천 개 일회성 임베딩: Gemini 비용 약 ~$0.05/이미지 → 약 $250. 자주 쓸 기능이면
  합리적, 실험용이면 부담.
- 캐시 무효화: 카탈로그에 새 에셋 추가되면 해당 id 만 임베딩 추가.

**결정 필요:** 현재 카테고리 기반 랜덤 노출로 충분한지 수개월 운영 후 판단.

### [16] 갤러리 낙서 / 어노테이션 (GalleryCanvas 위에 피드백 그리기)

**배경 (v1.10.31 기준):** 갤러리 캔버스(`F`) 에서 이미지 팬/줌 + 대표 지정까지 됨.
시안 비교 / 검토 시 "이 부분 다리 선이 이상함" 같은 포인팅 피드백을 남길 방법이 없음.
댓글로 텍스트 설명은 가능하지만 지점 특정이 어려움.

**옵션 A (추천) — 이미지 위 오버레이 낙서, 카드에 저장**
- 갤러리 타일마다 `🖊 낙서` 토글. 활성화 시 pointer drag 로 SVG path 그리기
- 저장: `card.data.annotations[]` — `{ image_url, actor, created_at, paths: [{ tool, color, points[] }] }`
- 팀원 모두 같은 낙서를 볼 수 있음 (코멘트 느낌), 레이어별 on/off 토글
- 도구: 펜(자유선), 화살표, 사각/원 하이라이트, 색상 3~4개, 지우기, 전체 지우기
- 공수: ~200줄. pointer event → SVG path 생성 + 저장 API

**옵션 B — 로컬 낙서만 (공유 안됨)**
- 세션 내에서만 보이고 저장 안함
- 단순 (~80줄), 팀 피드백엔 부족

**옵션 C — 낙서 → PNG 캡처해서 댓글 첨부**
- 캔버스에서 낙서 후 `📷 캡처` → 이미지+낙서 합성 PNG → `/api/upload` → 해당 카드 댓글에 첨부
- 기존 댓글 시스템 재사용 → 별도 스키마 불필요
- 공수: ~150줄 (Canvas 2D 합성)

**트레이드오프:**
- A 는 편집/삭제 가능, 실시간성 있음 but 스키마 추가 필요
- C 는 단순하지만 낙서 수정하려면 새로 그려서 올려야 함
- A+C 혼합 (낙서 저장 + 편의성으로 캡처 버튼) 도 가능

**프로필 연동:** 어떤 옵션이든 프로필 아이콘 + 이름으로 "누가 그렸는지" 표시.
여러 명이 겹쳐 그리면 레이어 구분.

**결정 필요:**
- 팀 협업 강도: 실시간 피드백이 주 목적이면 A, 캡처로 충분하면 C.
- 공수 여력: A ≈ 200줄 + PATCH 엔드포인트 재사용 / C ≈ 150줄 + 순수 프론트.
- 투표 기능과 시너지 가능 (시안 위에 의견 남기고 투표).

**Confluence 방향**

- **카드 → 페이지 push** (1순위)
  - 카드 상세 모달 / 리스트 우클릭에 "📤 위키에 공유"
  - 발행 단위: 카드 1:1 / 업데이트 태그 묶음 / 카테고리 카탈로그 중 선택
  - 본문: 제목·카테고리·스타일·크기·시안 갤러리·시트 4뷰·코멘트
  - 이미지: attachment 업로드 후 storage format 으로 inline
  - 재발행 = 같은 페이지 갱신 (Confluence 가 버전 자동 관리)
  - 카드 메타에 페이지 URL 저장 → 카드 ↔ 위키 양방향 링크
- **위키 → 카드 pull** (선택)
  - 특정 space / parent page / 라벨(CQL) 기반으로 스펙 페이지 가져와
    카드 description / 업데이트 태그 자동 채움

**Jira 방향**

- 카드 → Jira 이슈 자동 생성 (모델링팀 작업 티켓)
- 이슈 상태 변경 → 카드 stage 자동 동기 (모델링 완료 → ✅)
- 이슈 코멘트 ↔ 카드 코멘트 양방향 (선택)
- 카드 썸네일 = 이슈 첨부

**공통 요건**

1. **사내망 → `*.atlassian.net` HTTPS 아웃바운드 허용** (운영 PC 10.99.4.115 망 정책 확인이
   가장 큰 변수)
2. API 토큰 1개 (공용 봇 계정 권장) — 서버 `.env` 보관
3. 대상 space key / Jira project key / parent page / 라벨 규칙 합의

**예상 공수 (단계적)**

| 단계 | 범위 | 예상 |
|-----|------|------|
| 1 | Confluence 단방향 push (카드 1:1, 이미지 첨부) | 1~2일 |
| 2 | Jira 이슈 생성 + 상태 동기 | +1~2일 |
| 3 | Confluence pull / Jira 코멘트 양방향 sync | +3~5일 |

**결정 필요:**

- 1순위는 Confluence push — 외부 공유 갈증 해소.
- 망 정책 풀리는 시점에 1단계부터 진입. Jira 는 모델링팀 작업 흐름과 합쳐 검토 후 결정.
